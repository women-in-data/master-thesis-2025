# Stacking Pilot — Extended Analysis (Cross-Domain Ensemble)
**Авторы анализа:** Команда (Диана, Соня, Карина, Альбина)  
**Дата:** 2026-05-11  
**Скрипт:** `run_stacking.py`  
**Тестовая выборка:** team canonical split — 58 410 товаров, 4 643 контрафакта (7,95%)

---

## Контекст

Это расширенный пилотный анализ кросс-доменного ансамблирования.  
В отличие от предыдущего пилота Сони (`07_stacking_pilot_4domains.py`), который подбирал веса методом grid search по PR-AUC, данный анализ:

1. **Оценивает все фиксированные стратегии** — без оптимизации на тест-сете (честные оценки)
2. **Исследует феномен исключения доменов** — какие тройки/пары лучше четвёрки
3. **Применяет парный bootstrap** (n=1000) для статистической значимости
4. **Проверяет дополнительные варианты модели Сони** (e5_team, e7_team, e0_full)
5. **Исследует мета-классификаторы** (LR, CatBoost) на hold-out разбиении 50/50


## Методологические выводы

1. **Парадокс разнообразия:** при oracle recall gain 1.80× (K=1000), простое усреднение даёт лишь −0.0002 vs diana. Разнообразие на уровне топ-K не транслируется в улучшение R@P≥0.9 автоматически.

2. **Fragility R@P≥0.9:** bootstrap CI для R@P ширина ±0.04–0.08 против ±0.005–0.010 для PR-AUC — разрыв на порядок. Grid search «переоптимизирует» нестабильную метрику.

3. **Diana как якорь:** diana — сильнейшая модель по R@P≥0.9 (0.1154) и правильный фундамент ансамбля. Fusion добавляет поведенческий сигнал, albina добавляет глобальное ранжирование.

4. **Калибровка опасна для R@P≥0.9:** isotonic regression оптимизирует calibration loss, не rank loss в зоне P≥0.90. Применять до ансамблирования нельзя.

5. **Двойной вес FinTech:** 4way_fe5_d_a (fusion + e5 + diana + albina) неявно даёт FinTech-домену двойной вес. Это обосновано тем, что у Сони — наиболее богатая мультимодальная архитектура.
