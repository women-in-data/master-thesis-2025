# FinTech-домен: саммари для кросс-доменной главы

### 1. Цель и роль FinTech-домена в общем проекте

FinTech-домен — индивидуальный вклад в групповую ВКР по детекции контрафакта на Ozon, реализованный в подкаталоге `fintech_approaches/`. Специфика домена — поведенческие признаки продавца (доли возвратов, скорость продаж, GMV, время жизни товара и продавца), сильный дисбаланс классов (≈6.6% положительных) и требование к точечному предсказанию: метрика Recall при Precision≥0.9 (R@P≥0.9).

### 2. Единый сплит и методологическая инфраструктура

Для сопоставимости результатов между доменами зафиксирован единый сплит, воспроизводящий двойной `GroupShuffleSplit` по `SellerID` с `random_state=42` из `notebooks/baseline.ipynb` (cell 7). Размеры: train = 69 453 наблюдения / 3 908 продавцов / positive rate 6.10%; val = 69 335 / 3 908 / 6.02%; test = **58 410 / 3 351 / 7.95%**. Утечки `SellerID` между подвыборками отсутствуют (проверено assert-ами в `01_team_split.ipynb`); покрытие CLIP-эмбеддингов на test составляет 95.24%. Замороженные позиционные индексы и `y_test.npy` сохранены в `team_runs/splits/`. Это инфраструктурный вклад в общий проект: сплит — единая точка опоры для соавторов и для последующей кросс-доменной интеграции.

Конфигурация CatBoost в экспериментах team_runs/ идентична конфигурации индивидуальной FinTech-ветки (E0_canon, см. `fintech_experiment.ipynb` cell 5: `iterations=1000, depth=6, lr=0.05, eval_metric='AUC'`, `scale_pos_weight` по train, `early_stopping_rounds=50, random_seed=42`) и отличается от командной (`baseline.ipynb` cell 20: `iterations=700`, без early stopping). Этот выбор сделан намеренно: при фиксированной модели delta между экспериментами team_runs/ интерпретируются как эффект признакового пространства, а не смесь признаков и гиперпараметров. Прямое сопоставление с командной конфигурацией требует отдельного прогона `E0_team_native`, выходящего за рамки данной работы.

### 3. Серия экспериментов на едином сплите

На командном сплите проведены пять прогонов CatBoost с одной и той же конфигурацией (`iterations=1000, depth=6, lr=0.05, eval_metric='AUC', scale_pos_weight=15.40, early_stopping_rounds=50, random_seed=42`); варьируется только пространство признаков. Результаты на test (n = 58 410):

| Эксперимент | Признаки | ROC-AUC | PR-AUC | R@P≥0.9 |
|---|---|---:|---:|---:|
| E0_team_clean | командные 38 фич, без текста | 0.9050 | 0.5301 | 0.0022 |
| E0_team_full | мои 42 фичи, текст-как-category | 0.9366 | 0.6812 | 0.0691 |
| E5_team | мои + CLIP-PCA-25 | 0.9508 | 0.7189 | 0.0950 |
| E7_team | мои + e5-small text-PCA-25 | 0.9521 | 0.7224 | 0.0915 |
| **Fusion_team** | мои + CLIP-PCA-25 + text-PCA-25 | **0.9522** | **0.7284** | **0.1077** |

Все модели сошлись по early stopping (best_iteration < 1000).

### 4. Главный методологический результат

**(а) Текст в табличной модели.** Включение четырёх текстовых полей как category-признаков (E0_team_full vs E0_team_clean) увеличивает R@P≥0.9 с 0.0022 до 0.0691 (Δ = +0.0669) и PR-AUC с 0.5301 до 0.6812 (Δ = +0.151). Низкое значение R@P≥0.9 у командного pure-tabular baseline (0.0022 ≈ 5 положительных объектов на 4 642 в test) указывает, что без текстовой модальности задача детекции контрафакта в данной формулировке практически нерешаема, а добавление текстовых полей восстанавливает работоспособность модели на этой метрике. Прирост статистически значим: 95% paired bootstrap CI (n = 1000, seed = 42) равны [+0.020, +0.144] для R@P≥0.9 и [+0.140, +0.162] для PR-AUC.

**(б) Воспроизводимость эффекта мультимодальности.** На индивидуальном сплите (test = 34 416) добавление CLIP-PCA-25 поверх табличной модели поднимает R@P≥0.9 с 0.1721 (E0_canon) до 0.2956 (E5); на командном сплите (test = 58 410) аналогичное добавление поднимает R@P≥0.9 с 0.0691 (E0_team_full) до 0.0950 (E5_team). Абсолютные значения R@P≥0.9 на командном сплите примерно втрое ниже (PR-AUC, напротив, на командном сплите выше: 0.7189 против 0.6724 на индивидуальном). Это согласуется с известной чувствительностью thresholded recall к составу test set: командный test содержит 3 351 неизвестного продавца и положительный класс плотностью 7.95% против 1 676 / 5.93% на индивидуальном сплите, что объективно усложняет достижение precision ≥ 0.9. Направление эффекта при добавлении мультимодальных эмбеддингов сохраняется на обоих сплитах. Прирост по PR-AUC значим на 95% bootstrap CI (E5_team vs E0_team_full: [+0.033, +0.043]; E7_team vs E0_team_full: [+0.035, +0.047]); прирост по R@P≥0.9 направленный, но границы CI пересекают ноль — R@P≥0.9 является высокодисперсной хвостовой метрикой, основной показатель — PR-AUC.

**(в) Объединение модальностей.** Конкатенация CLIP- и text-эмбеддингов в одной модели (Fusion_team) даёт лучшие метрики по всем трём показателям: ROC-AUC = 0.9522, PR-AUC = 0.7284, R@P≥0.9 = 0.1077. Прирост PR-AUC над лучшей одномодальной моделью статистически значим: Fusion_team vs E5_team даёт Δ = +0.0094 (95% CI [+0.005, +0.014]); Fusion_team vs E7_team даёт Δ = +0.0061 (95% CI [+0.001, +0.011]). По R@P≥0.9 прирост направленный (Fusion_team vs E5_team: Δ = +0.0097), но границы CI пересекают ноль. Значимый прирост по PR-AUC при малой абсолютной величине свидетельствует о комплементарности визуальной и текстовой модальностей: они несут не идентичный, а дополняющий друг друга сигнал.

### 5. Готовые артефакты для кросс-доменной интеграции

Для соавторов подготовлены и зафиксированы (ветка `claude/dazzling-vaughan`):

- `team_runs/splits/team_{train,val,test}_idx.npy`, `y_test.npy` — единый сплит, воспроизводимый из `ozon_train.csv`;
- `team_runs/embeddings/text_e5_small.parquet` — текстовые эмбеддинги `intfloat/multilingual-e5-small` (197 198 × 384), переиспользуемые;
- `team_runs/proba/test_proba_{e0_team_clean,e0_team_full,e5_team,e7_team,fusion_team}.npy` — пять массивов вероятностей на едином test;
- `team_runs/results/team_split_results.md` — полный технический отчёт с конфигурациями, метриками и bootstrap-интервалами.

Для построения кросс-доменного stacking-ансамбля требуются probability arrays соавторов на том же командном test (n = 58 410). Технический протокол интеграции: (1) загрузить общие индексы из `team_runs/splits/team_{train,val,test}_idx.npy`; (2) обучить модель домена на объединённой выборке `train_idx ∪ val_idx` с использованием val для подбора гиперпараметров; (3) сохранить `predict_proba` на `test_idx` как массив формы `(58410,)` в файл вида `test_proba_{domain}.npy`. После сбора четырёх массивов кросс-доменный stacking реализуется как мета-классификатор (LogisticRegression или CatBoost) над выходами доменных моделей. Эта задача относится к общему этапу интеграции и выходит за рамки FinTech-домена.
